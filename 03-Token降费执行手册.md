很多用户觉得自己“只说了一两句话”，但账单里的 input tokens 却突然变成十几万。常见原因不是这一句话很长，而是当前对话已经积累了很长的历史上下文：每次发送新消息时，客户端可能会把当前消息和此前 Messages 上下文一起发给模型。

如果这时再使用 Opus 这类单价更高的模型，费用会被进一步放大。降低账单的关键不是少说话，而是减少每轮请求携带的历史上下文，并把高单价模型用在真正需要的任务上。

## 先看账单，不靠感觉猜

打开用量页面：

<https://v3.codesome.cn/usage>

建议先做 3 步：

1. 把时间范围筛选为当天。

2. 点击右侧导出 CSV。

3. 重点看 `Input Tokens`、`Output Tokens`、`Cache Read Tokens`、`Cache Creation Tokens`、`Model` 和 `Rate Multiplier`。

账单可以先用这个公式理解：

```plain
最终扣费 = Token 使用量 × 模型原价 × 当前分组倍率
```

更完整地说，输入、输出、缓存读取、缓存写入都会计费：

```plain
Billed Cost =
(
  Input Tokens × Input 原价
  + Output Tokens × Output 原价
  + Cache Read Tokens × Cache Read 原价
  + Cache Creation Tokens × Cache Creation 原价
) ÷ 1,000,000 × Rate Multiplier
```

所以费用不是凭空出现的。排查时主要看三件事：Token 量是不是异常、模型是不是高单价、当前分组倍率是多少。

## 四个高费用信号

| 信号                | 说明                     |
| ----------------- | ---------------------- |
| 单轮 input 接近 10 万  | 当前对话历史太长，每轮都在携带大量上下文   |
| Cache Creation 很高 | 系统在反复创建大缓存，长上下文被重新写入   |
| Cache Read 量巨大    | 缓存读取单价低，但量级大了依然会产生明显费用 |
| 使用 Opus 或高倍率分组    | 长上下文叠加高单价模型，费用上涨更明显    |

很多人误以为“缓存很便宜，所以不用管”。缓存确实比普通输入输出便宜，但它不是免费。几十万、几百万甚至上千万 tokens 的缓存读写，再乘上模型原价和分组倍率，最后仍然会很贵。

## 为什么不是只算刚输入的那句话

连续对话里，每次请求通常不是只发送你刚输入的新消息，而是会带上当前对话里的历史上下文。历史越长，每一轮请求的 input tokens 就越高。

所以会出现这种现象：

| 你的感受      | 实际计费原因                         |
| --------- | ------------------------------ |
| 我只聊了几句    | 系统同时处理了此前大量对话历史                |
| 我输入只有一两句话 | input tokens 可能包含 Messages 上下文 |
| 前几天费用没这么高 | 前几天上下文较短，或主要使用 Sonnet          |
| 今天突然很贵    | 今天上下文变长，同时使用 Opus 或高倍率分组       |

这不是某一句话导致的，而是“长上下文 + 高单价模型 + 连续多轮对话”叠加后的结果。

## 省钱动作一：长任务常压缩

如果还想继续当前对话，不想马上新开会话，优先使用压缩命令：

```plain
/compact
```

建议节奏：

```plain
每完成 4 轮对话，第 5 轮先不要继续提新问题，先输入 /compact
```

如果有特别想保留的内容，可以这样写：

```plain
/compact 请保留产品设计结论、运营策略、文章选题方向和已经确认的执行步骤
```

这样压缩会更有侧重，不只是把上下文变短，也能尽量保留后续真正需要的信息。

## 省钱动作二：任务结束就新开

一个上下文不要反复用。任务完成、阶段切换、话题切换，建议直接新开对话：

```plain
/new
```

适合新开的场景：

| 场景          | 建议                       |
| ----------- | ------------------------ |
| 一篇文章已经写完    | 新开对话做下一篇                 |
| 产品设计问题已经讨论完 | 新开对话处理运营问题               |
| 话题从创作切到数据分析 | 新开对话                     |
| 当前对话已经很长    | 先 `/compact`，再视情况 `/new` |

不要对旧对话产生感情。旧对话越长，后面每一句的上下文成本越容易变高。

## 省钱动作三：按任务切换模型

不同任务不一定都要用最高单价模型。在 Claude Code 里可以输入：

```plain
/model
```

推荐规则：

| 任务类型             | 推荐模型策略          |
| ---------------- | --------------- |
| 简单信息整理、摘要、改写     | 优先用 Sonnet      |
| 日常运营问题、普通方案讨论    | 优先用 Sonnet      |
| 复杂文案、深度推理、复杂数据分析 | 再切到 Opus        |
| 已经是长上下文的连续对话     | 尽量避免继续用 Opus 硬聊 |

核心原则是：不要把所有任务都交给高单价模型。简单任务用更经济的模型，复杂任务再使用更强模型。

## 省钱动作四：延续旧内容时，优先 export

如果新开了对话，但还想继续之前的工作，不建议默认使用：

```plain
/resume
```

因为通过 `/resume` 继续旧对话时，原来的缓存可能会失效，系统需要重新构建长上下文。你在 resume 后只聊一句，也可能因为重建上下文而让账单变高。

更稳妥的做法是：

1. 输入 `/export` 导出当前对话。

2. 新开窗口或新开对话。

3. 让 AI 读取导出的本地文件，基于里面的上下文继续。

如果只是延续任务，而不是恢复完整历史，优先 `/export` 或一段精简摘要；只有确实需要恢复完整旧对话时，再用 `/resume`。

## 省钱动作五：问题一次说清

问题说不清，模型就会猜。猜错了你再补充，补充完它再重新判断，这些来回都会消耗 token。

新任务可以用这个模板：

```plain
我的目标是：
我的系统环境是：
我使用的客户端是：
我已经完成的步骤是：
当前遇到的问题是：
我希望你输出的形式是：
请先给我最短可执行方案，不要展开长篇解释。
```

报错排查可以用这个模板：

```plain
我遇到了一个报错，请你帮我排查。

目标：
环境：
客户端：
模型/分组：
已经做过：
报错内容：

请你按可能性从高到低排序。
每次只给我一个排查步骤。
如果信息不足，请先问我最关键的一个问题。
```

这样模型不需要先猜一轮，可以直接进入排查，输出也会更短、更准。

## 省钱动作六：保存结果，别反复重做

模型已经给了配置命令、排查步骤、文档内容、代码片段，就要及时保存。能保存到本地文件就保存到本地文件，能写进项目 README 就写进 README，能放进飞书文档就放进飞书文档。

阶段完成时，可以让模型输出一段可保存总结：

```plain
请把当前阶段整理成一段可保存总结，包含：

1. 已完成什么
2. 当前结论是什么
3. 关键命令/配置是什么
4. 还剩什么问题
5. 下一步做什么

要求简短、清楚，方便我复制到文档里。
```

下次继续时，带这段总结就够了，不要把整段历史重新复制进去。

## 让 AI 帮你分析账单

如果你已经导出了 CSV，可以直接把文件路径给 AI，用这个精简提示词：

```plain
请读取并分析这个 API 账单 CSV：

CSV 文件路径：<把你的 CSV 文件路径放这里>

已知前提：模型公开单价和当前分组倍率已经固定，不要从账单金额反推模型价格。必要时先联网查询官方价格页，确认常见模型（例如 GPT-5.6 Terra、GPT-5.6 Sol、Claude Opus、Claude Sonnet、xAI 相关模型）的 input、output、cache read、cache creation / cache write 单价；再结合账单里的模型名、倍率和 token 用量核算费用。GPT-5.6 配置优先填写精确模型 ID，避免裸 `gpt-5.6` 默认落到更贵的 Sol。

请按这个顺序输出：
1. 用“最终扣费 = 各类 Token 使用量 × 对应公开单价 × 当前分组倍率”解释扣费逻辑，并说明 input、output、cache read、cache creation / cache write 不能混在一起算。
2. 不要反推模型单价；直接列出本次分析采用的公开单价、分组倍率和来源/查询时间。
3. 选择 Billed Cost 最高的一条请求，按 input、output、cache read、cache creation / cache write 分项验算原始费用和最终扣费。
4. 统计全量 Input、Output、Cache Read、Cache Creation / Cache Write tokens，并计算总费用。
5. 判断主要费用来自普通 input、output、cache read 还是 cache creation / cache write。
6. 重点分析为什么 Token 占用高：是否因为上下文过长、历史对话被反复带入、工具调用/文件内容太大、缓存写入反复重建、输出过长，还是同一任务多次重跑。
7. 给出 3 条可执行的降本建议，每条都要对应到一个具体的高 token 成因。
8. 用一句话总结：在单价和倍率固定的情况下，哪一种 Token 用量异常把总成本推高。

要求：不要只说 tokens 很多，要写具体数字；不要只说费用很高，要写具体美元金额；不要把价格异常当成主要结论，除非公开单价或倍率确实填错。
```

这样可以快速判断：这次贵到底是因为普通输入太高、输出太长、缓存读取量巨大，还是缓存写入反复重建。

## 最省钱的日常工作流

可以记住这套流程：

1. 每个独立任务尽量单独开对话。

2. 同一个任务连续聊 4 轮后，先 `/compact` 再继续。

3. 简单整理、摘要、运营建议优先用 Sonnet。

4. 复杂创作、深度分析、关键决策再切到 Opus。

5. 每天查看一次用量页面，重点看 input tokens 和 cache creation 是否异常升高。

6. 一旦单轮 input tokens 接近 10 万，立刻压缩或新开对话。

7. 延续旧内容优先 `/export`，不得已再 `/resume`。

最后记住一句话：

```plain
短任务新开，长任务常压缩；简单任务用 Sonnet，复杂任务再用 Opus；延续任务先 export，不得已再 resume。
```

把这些动作变成习惯，通常就能把异常高的 input tokens 和缓存重建拉下来，避免“只聊几句却扣很多钱”的情况反复出现。
